CocoaPods教程

发表于 2015-06-17   |   分类于 CocoaPods   |  

英语好的就直接看官方教程吧。

奈何自己英语不是很好，所以翻译得不是很好，随便拍砖，都接着。:)

当你开发iOS应用时，会经常使用到很多第三方开源类库，可能某个类库又用到其他类库，所以要使用它，必须得另外下载其他类库，而其他类库又用到其他类库，手动一个个去下载所需类库十分麻烦。另外一种常见情况是，你项目中用到的类库有更新，你必须得重新下载新版本，重新加入到项目中，十分麻烦。所以我们需要CocoaPods。

什么是CocoaPods

CocoaPods是一个用来帮助我们管理第三方依赖库的工具。
我们项目中所有的依赖都写在一个叫Podfile的文本文件里面，CocoaPods会帮我们解决各个第三方库之间的依赖，获取库最新的源码，然后将它们link在一起创建一个Xcode的workspace来构建我们的项目。

使用CocoaPods的目的是为了让我们能够更集中的管理第三方开源库。

Getting Started

安装

CocoaPods是使用Ruby实现的，OS X系统已经默认安装了Ruby。然后我们就可以通过以下命令来安装CocoaPods了：

$ sudo gem install cocoapods

如果你在天朝，而你又没有梯子，那么你会发现这个命令执行地非常非常慢，因为有一堵墙非常高，非常厚，以至于很多网站都不能正常访问，有没有办法解决这个问题而不用自己备梯子呢？当然有！我们可以用淘宝的Ruby镜像来安装CocoaPods。按照下面的顺序在终端中敲入依次敲入命令：

$ gem sources --remove https://rubygems.org/
// 等有反应之后再敲入以下命令
$ gem sources -a http://ruby.taobao.org/

为了验证你的Ruby镜像是并且仅是taobao，可以用以下命令查看：

$ gem sources -l

只有在终端中出现下面文字才表明你上面的命令是成功的：

*** CURRENT SOURCES ***

http://ruby.taobao.org/

成功之后再安装CocoaPods就没有那么慢了。

非root用户安装

也就是不使用sudo，如果你不想让RubyGems此次安装使用管理员权限，你可以将RubyGems安装到你的用户目录下面，有2个办法：

  1. gem install加一个flag：—user-install
  2. 配置RubyGems的环境

官方认为第二个办法相对更好一点，要配置RubyGems的环境，也很简单，打开Home目录下的.profile文件（需要系统显示隐藏文件，方法Google或者度娘以下就可以找到），如果没有该文件，则创建一个，添加或者修改下面几行代码：

export GEM_HOME=$HOME/.gem
export PATH=$GEM_HOME/bin:$PATH

注意啦，如果你选择第一种方式，还是需要配置你的.profile文件，设置PATH，或者在完整路径下使用命令行安装。你可以通过以下命令来找到gem的安装目录：

$ gem install cocoapods --user-install
$ gem which cocoapods
/Users/eloy/.gem/ruby/2.0.0/gems/cocoapods-0.29.0/lib/cocoapods.rb
$ /Users/eloy/.gem/ruby/2.0.0/bin/pod install

更新CocoaPods

再次用gem安装CocoaPods就可以更新CocoaPods了：

$ [sudo] gem install cocoapods

或者也可以安装pre-release版本：

$ [sudo] gem install cocoapods --pre

如果你原来安装CocoaPods的时候使用了sudo，那么你应该要加上sudo。

Using CocoaPods

The Podfile

Podfile是啥

Podfile是一个用来声明你所有使用的第三方库的文件，没有文件后缀。Podfile默认会创建一个名为default的target，link到你项目的第一个target。

Podfile内容可以非常简单：

pod 'AFNetworking', '~> 2.0'

也可以更复杂：

source 'https://github.com/CocoaPods/Specs.git'
source 'https://github.com/Artsy/Specs.git'

platform :ios, '8.0'
inhibit_all_warnings!

xcodeproj 'MyProject'

pod 'ObjectiveSugar', '~> 0.5'
pod 'Artsy+UILabels', '~> 1.0'

target :test do
    pod 'OCMock', '~> 2.0.1'
end

post_install do |installer|
    installer.project.targets.each do |target|
        puts target.name
    end
end

如果你想要多个target之间共用一个pods，比如Tests target，你可以这么写：

platform :osx, '10.7'

link_with 'MyApp', 'MyApp Tests'
pod 'AFNetworking', '~> 2.0'
pod 'Objection', '0.9'

有时候一些库编译会有警告，如果不想看到这些警告，可以在platform :xxx, 'x.x'后面添加这个：

inhibit_all_warnings!

这样就不会再看到警告了。

但是有些库如果是在上面这种关闭警告的方式下，会编译失败，比如ReactiveCocoa，所以需要单独将这个库的编译警告开启：

pod 'ReactiveCocoa', '~> 2.1.8', :inhibit_warnings => true

指定pod的版本

当你想在一个项目里面使用一个pod的最新版本，你可以删掉这个pod的版本约束：

pod 'SSZipArchive'

如果你还想使用一个pod的某个特定的版本，你可以指定这个pod的版本：

pod 'Objection', '0.9'

除了pod没有版本，或者指定了一个版本，那么可以使用逻辑操作符：

  • ‘> 0.1’ 任何高于0.1的版本
  • ‘>= 0.1’ 版本为0.1或者更高
  • ‘< 0.1’ 任何低于0.1的版本
  • ‘<= 0.1’ 版本为0.1或者更低

此外，CocoaPods除了逻辑操作符，还有一个更优的操作符”~>”：

  • ‘~> 0.1.2’ 版本为0.1.2或者最高到0.2，不包括0.2或者更高的版本，相当于[0.1.2, 0.2)
  • ‘~> 0.1’ 版本为0.1或者最高到1.0，不包括1.0或者更高的版本，相当于[0.1, 1.0)
  • ‘~> 0’ 版本为0或者更高，这个就跟不指定版本一样

想了解更多内容，请看：

  • Semantic Versioning
  • RubyGems Versioning Policies

版本冲突

Pods经常会依赖其他pods，这样就很容易产生冲突，比如如下的错误信息：

[!] Podfile tries to activate `GoogleAnalytics-iOS-SDK (= 2.0beta4)', but already activated version `3.0' by ARAnalytics/GoogleAnalytics (1.6).

解决这个冲突也很简单，只要将GoogleAnalytics-iOS-SDK pod写在ARAnalytics之前并指定版本就OK啦：

pod 'GoogleAnalytics-iOS-SDK', '2.0beta4'
pod 'ARAnalytics/GoogleAnalytics'

使用本地目录

pod 'AFNetworking', :path => '~/Documents/AFNetworking'

使用这个option，CocoaPods会设置这个文件夹为Pod的根目录，在Pods项目中会直接从这里link文件。这意味着你所有的编辑在CocoaPods安装中都会被保存(This means that your edits will persist between CocoaPods installations.)。这个目录可以是你从你喜欢的源码版本控制(SCM)中检出(checkout)的，或者也可以是当前repo的中的一个git submodule。
注意，Pod文件的podspec也应该要在这个目录下。

From a podspec in the root of a library repo

有时候你可能想要用一个Pod的bleeding edge版，一个特定的修改版，或者你自己的fork，如果是这样，你可以指定你的pod的声明。
使用这个repo的master分支：

pod 'AFNetworking', :git => 'https://github.com/gowalla/AFNetworking.git'

使用这个repo的其他分支：

pod 'AFNetworking', :git => 'https://github.com/gowalla/AFNetworking.git', :branch => 'dev'

使用这个repo的某个tag：

pod 'AFNetworking', :git => 'https://github.com/gowalla/AFNetworking.git', :tag => '0.7.0'

或者指定一个commit：

pod 'AFNetworking', :git => 'https://github.com/gowalla/AFNetworking.git', :commit => '082f8319af'

然而，需要特别注意的是，这意味着这个版本需要满足这个Pod与其他Pods的依赖关系。
podspec文件应该在这个repo的根目录下，如果这个library在它的repo中还没有podspec文件，你可以使用下面的章节中列出的方法中的一个。

添加Pods到Xcode项目中

开始添加之前，需要注意以下两点：

  1. 查看Specs repository或者cocoapods.org确认有你想要使用的第三方开源库
  2. 你的电脑上已经安装了CocoaPods

安装

创建一个Podfile文件，然后添加你的依赖pods：

pod 'AFNetworking', '~> 2.0'
pod 'ObjectiveSugar', '~> 0.5'
  • 在你的项目目录下运行命令：$ pod install
  • 打开 App.xcworkspace 然后 build

创建一个新的CocoaPods Xcode项目

步骤很简单：

  • 就像你平常那样创建一个新的Xcode项目
  • 打开terminal(终端)，$ cd 你的项目目录
  • 创建一个Podfile文件，可以使用命令$ touch Podfile或者vim Podfile
  • 如果你是用touch命令创建的，你还需要再执行vim Podfile来在命令行下打开Podfile文件，所以上一步建议使用vim Podfile来创建并在命令行下打开Podfile。

  • 第一行需要指定平台和支持的系统版本

    platform :ios, ‘8.0’

  • 添加一个指定名字的CocoaPod，pod '$PODNAME'

    pod ‘ObjectiveSugar’

  • 保存Podfile，vim下可以在编辑状态下按一下Esc，然后输入:wq回车就可以保存修改了。

  • 运行命令$ pod install
  • 打开刚刚生成的 MyApp.xcworkspace 文件，以后编写这个app都需要使用这个文件。

整合一个已经存在的workspace

将CocoaPods与一个本地的workspace整合在一起需要额外的在Podfile里添加一行代码，很简单，指定这个.xcworkspace的文件名就行了，比如：

workspace 'MyWorkspace'

是否需要将Pods目录添加到source control？

这个随便你，官方建议是将Pods目录添加到source control。

添加到source control的好处

  • clone repo之后，就算没有安装CocoaPods，项目也可以直接build和运行。也不需要运行命令pod install，更不需要联网。
  • Pod artifacts(code/libraries)一直都是可用的，就算这个Pod的源码(比如在Github上)不在了。
  • The Pod artifacts are guaranteed to be identical to those in the original installation after cloning the repo.

忽略Pods目录的好处

  • 源码库会比较小，占用的空间也小
  • 只要所有Pods的源码(比如在Github上)都是可用的，CocoaPods一般都是能够重建相同的安装(CocoaPods is generally able to recreate the same installation.)。(技术上来说当在Podfile文件中没有使用一个commit SHA，不保证运行pod install命令会去获取并重建相同的artifacts)。
  • 当执行源码控制操作的时候不会有任何的冲突，比如合并不同的Pod版本的分支。

不论你是否同步(添加到SCM)Pods目录，Podfile和Podfile.lock文件都应该是在版本控制下的，也就是说，都是添加到版本控制中的。

Podfile.lock是啥

这个文件是在第一次运行pod install命令之后自动生成的，跟踪记录每一个已经安装的Pod的版本。比如，假如有以下的dependency指定在Podfile中：

pod 'RestKit'

运行pod install命令将会安装RestKit的当前版本，因为有一个Podfile.lock被生成，会指定一个正确的安装版本(比如RestKit 0.10.3)，多亏了这个文件，在这个模拟的项目上运行pod install命令和之后在别的机器上安装的都还是RestKit 0.10.3，就算RestKit有新的版本可用。CocoaPods会遵守Podfile.lock中的Pod版本，除非Podfile中的dependency更新了或者运行了pod update命令(因为生成了一个新的Podfile.lock)，如此CocoaPods避免了unexpected changes to dependencies这个让人头疼的事情。

这个文件应该一直都是在版本控制下的(This file should always be kept under version control.)。

这背后都发生了什么

在Xcode中，with references directly from the ruby source, it：

  1. 创建或者更新workspace
  2. 如果需要，添加你的project到workspace
  3. 如果需要，添加CocoaPods静态库project到workspace
  4. 添加libPods.a到targets => build phases => link with libraries
  5. 添加CocoaPods Xcode配置文件到你App的project中
  6. 将你App的target configurations改为基于CocoaPods的
  7. 添加一个拷贝所有你安装的pods资源的build phase到你的app bundle。也就是说，在所有其他的build phases之后有如下一个”Script build phase”：
    • Shell: /bin/sh
    • Script: ${SRCROOT}/Pods/PodsResources.sh

注意在第三步之前，如果CocoaPods静态库已经在在你的project里了，将会跳过。This is largely based on Jonah Williams’ work on Static Libraries.

Pods and Submodules

CocoaPods和git submodules都尝试解决非常类似的问题。都是努力去简化在项目中包含第三方代码的处理。Submodules是link一个项目的指定commit，而一个CocoaPod约束一个versioned developer release。

从submodules迁移到CocoaPods

在你决定完全迁移到CocoaPods之前，确认一下所有你使用的库是可用的。这也是记录你当前使用的库的版本的好办法，然后你就可以安装CocoaPods，使用相同的库。It’s also a good idea to do this incrementally, going dependency by dependency instead of one big move.

  1. 安装CocoaPods，如果你还没有安装的话
  2. 创建你的Podfile文件
  3. 移除submodule的引用
  4. 在你的Podfile中添加一个刚刚移除的submodule库的引用
  5. 运行pod install
