当你开发 iOS 应用时，会经常使用到很多第三方开源类库，可能某个类库又用到其他类库，所以要使用它，必须得另外下载其他类库，而其他类库又用到其他类库，手动一个个去下载所需类库十分麻烦。另外一种常见情况是，你项目中用到的类库有更新，你必须得重新下载新版本，重新加入到项目中，十分麻烦。所以我们需要 CocoaPods。

什么是 CocoaPods

CocoaPods 是一个用来帮助我们管理第三方依赖库的工具。
我们项目中所有的依赖都写在一个叫 Podfile 的文本文件里面，CocoaPods 会帮我们解决各个第三方库之间的依赖，获取库最新的源码，然后将它们link在一起创建一个 Xcode 的 workspace 来构建我们的项目。

使用 CocoaPods 的目的是为了让我们能够更集中的管理第三方开源库。

除了 pod 没有版本，或者指定了一个版本，那么可以使用逻辑操作符：

  • ‘> 0.1’ 任何高于0.1的版本
  • ‘>= 0.1’ 版本为0.1或者更高
  • ‘< 0.1’ 任何低于0.1的版本
  • ‘<= 0.1’ 版本为0.1或者更低

此外，CocoaPods 除了逻辑操作符，还有一个更优的操作符 “~>”：

  • ‘~> 0.1.2’ 版本为0.1.2或者最高到0.2，不包括0.2或者更高的版本，相当于[0.1.2, 0.2)
  • ‘~> 0.1’ 版本为0.1或者最高到1.0，不包括1.0或者更高的版本，相当于[0.1, 1.0)
  • ‘~> 0’ 版本为0或者更高，这个就跟不指定版本一样

想了解更多内容，请看：

  • Semantic Versioning
  • RubyGems Versioning Policies

是否需要将 Pods 目录添加到 source control ？

这个随便你，官方建议是将 Pods 目录添加到 source control。

添加到 source control 的好处

  • clone repo 之后，就算没有安装 CocoaPods，项目也可以直接 build 和运行。也不需要运行命令pod install，更不需要联网。
  • Pod artifacts(code/libraries) 一直都是可用的，就算这个 Pod 的源码(比如在Github上)不在了。
  • The Pod artifacts are guaranteed to be identical to those in the original installation after cloning the repo.

忽略 Pods 目录的好处

  • 源码库会比较小，占用的空间也小
  • 只要所有 Pods 的源码(比如在Github上)都是可用的，CocoaPods 一般都是能够重建相同的安装 (CocoaPods is generally able to recreate the same installation.)。(技术上来说当在 Podfile 文件中没有使用一个 commit SHA，不保证运行pod install命令会去获取并重建相同的 artifacts)。
  • 当执行源码控制操作的时候不会有任何的冲突，比如合并不同的 Pod 版本的分支。

不论你是否同步 (添加到SCM)Pods 目录，Podfile 和 Podfile.lock 文件都应该是在版本控制下的，也就是说，都是添加到版本控制中的。

Podfile.lock 是啥

这个文件是在第一次运行pod install命令之后自动生成的，跟踪记录每一个已经安装的 Pod 的版本。比如，假如有以下的 dependency 指定在 Podfile 中：

运行pod install命令将会安装 RestKit 的当前版本，因为有一个 Podfile.lock 被生成，会指定一个正确的安装版本(比如RestKit 0.10.3)，多亏了这个文件，在这个模拟的项目上运行pod install命令和之后在别的机器上安装的都还是 RestKit 0.10.3，就算 RestKit 有新的版本可用。CocoaPods 会遵守 Podfile.lock 中的 Pod 版本，除非 Podfile 中的 dependency 更新了或者运行了pod update命令(因为生成了一个新的 Podfile.lock)，如此 CocoaPods 避免了 unexpected changes to dependencies这 个让人头疼的事情。

这个文件应该一直都是在版本控制下的 (This file should always be kept under version control.)。

这背后都发生了什么

在 Xcode 中，with references directly from the ruby source, it：

  1. 创建或者更新 workspace
  2. 如果需要，添加你的 project 到 workspace
  3. 如果需要，添加 CocoaPods 静态库 project 到 workspace
  4. 添加 libPods.a 到 targets => build phases => link with libraries
  5. 添加 CocoaPods Xcode 配置文件到你App的project中
  6. 将你App的 target configurations 改为基于 CocoaPods 的
  7. 添加一个 拷贝所有你安装的 pods 资源的 build phase 到你的 app bundle。也就是说，在所有其他的 build phases 之后有如下一个 “Script build phase”：
    • Shell: /bin/sh
    • Script: ${SRCROOT}/Pods/PodsResources.sh

注意在第三步之前，如果 CocoaPods 静态库已经在在你的 project 里了，将会跳过。This is largely based on Jonah Williams’ work on Static Libraries.

Pods and Submodules

CocoaPods 和 git submodules 都尝试解决非常类似的问题。都是努力去简化在项目中包含第三方代码的处理。Submodules 是 link 一个项目的指定 commit，而一个 CocoaPod 约束一个 versioned developer release。

从 submodules 迁移到 CocoaPods

在你决定完全迁移到 CocoaPods 之前，确认一下所有你使用的库是可用的。这也是记录你当前使用的库的版本的好办法，然后你就可以安装 CocoaPods，使用相同的库。It’s also a good idea to do this incrementally, going dependency by dependency instead of one big move.

  1. 安装 CocoaPods，如果你还没有安装的话
  2. 创建你的 Podfile 文件
  3. 移除 submodule 的引用
  4. 在你的 Podfile 中添加一个刚刚移除的 submodule 库的引用
  5. 运行pod install
